# Feature Design — Automatic Per-Directory "Complete" Flag

> Design spec for adding an automatic, DB-only completeness flag for every
> archived directory: did **all** the files the scan found under a directory end
> up either archived to tape or explicitly recorded as a known omission? No tape
> reads. Written 2026-07-11. Status: **design approved, not yet implemented.**

---

## 1. Problem

Small files (< `index_min_file_mb`, ~99 % of files here) are packed into bundle
ZIPs and **not** given individual `files_index` rows — they live only in the
per-bundle aggregate (`directory_archive_bundles`) and in the on-tape manifest.
Per-file fetch status (`remote_file_state`) is **transient**: it is deleted when
a chunk reaches `done` (except `source_missing`). So after a run there is no
durable answer to *"was this directory backed up in full?"* — only what was
archived is recorded, never compared against what should have been there.

The existing `validate_directory_catalog` checks **internal consistency**
(bundle totals == directory-tree totals) but never compares against the **source
scan**. There is no independent per-directory completeness signal.

## 2. Key idea — reconciliation, entirely from the DB

All three inputs already exist in PostgreSQL; nothing needs to touch the tape:

```
expected (what the scan found)   remote_snapshot_files   -> files/bytes per dir
archived (what was written)      directory_tree_index    -> files/bytes packed
missing  (known omissions)       source_missing state    -> found but unfetchable
────────────────────────────────────────────────────────────────────────────
is_complete[dir] = (archived_direct + source_missing_direct) >= expected_direct
```

A directory is **complete** when every file the scan found directly in it was
either archived or explicitly accounted as `source_missing` (an accepted,
recorded omission — not a silent gap).

## 3. Three design decisions (and why)

### 3.1 A new durable table `directory_completeness` — not a flag on `directory_tree_index`
`directory_tree_index` is **per-bundle**: a directory split across N chunks has N
rows. A flag there would fragment across those rows. A dedicated table gives
**one clean row per directory per session**.

```sql
-- scripts/sql/010_postgres_directory_completeness.sql
CREATE TABLE IF NOT EXISTS directory_completeness (
    completeness_id       BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    source_host           TEXT NOT NULL,
    original_dir_path     TEXT NOT NULL,
    tape_label            TEXT REFERENCES tapes(volume_label)
        ON UPDATE CASCADE ON DELETE CASCADE,
    remote_session_id     BIGINT REFERENCES remote_sessions(session_id)
        ON DELETE CASCADE,
    local_session_id      BIGINT REFERENCES local_sessions(session_id)
        ON DELETE CASCADE,
    expected_files        BIGINT NOT NULL CHECK (expected_files >= 0),
    expected_bytes        BIGINT NOT NULL CHECK (expected_bytes >= 0),
    archived_files        BIGINT NOT NULL CHECK (archived_files >= 0),
    archived_bytes        BIGINT NOT NULL CHECK (archived_bytes >= 0),
    source_missing_files  BIGINT NOT NULL DEFAULT 0 CHECK (source_missing_files >= 0),
    is_complete           BOOLEAN NOT NULL,
    verified_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (source_host, original_dir_path, remote_session_id)
);
CREATE INDEX IF NOT EXISTS idx_dir_completeness_incomplete
    ON directory_completeness(remote_session_id)
    WHERE is_complete = FALSE;
CREATE INDEX IF NOT EXISTS idx_dir_completeness_path
    ON directory_completeness(source_host, original_dir_path);
```
`CREATE TABLE IF NOT EXISTS` is **safe on a live DB** — it adds a table and takes
no lock on the existing catalog tables, so it will not disturb a running archive.

### 3.2 Compute at **session completion**, not per chunk
A directory can span multiple chunks, so "complete" is unknowable until every
chunk touching it is `done`. The reconciliation therefore runs **once, when the
session is marked `completed`**.

**Critical ordering:** it must run **before** `cleanup_unreferenced_remote_data`,
because that cleanup deletes `remote_snapshot_files` — the source of `expected`.
Snapshot data currently survives until that maintenance op is run manually, but
the reconcile stores its result durably, so the flag outlives later cleanup.

### 3.3 Compare on **direct** counts, not recursive
`recursive_*` double-counts across tree levels. Using `direct_*` (files directly
in a directory, excluding sub-directories) lets each directory be checked
independently; the whole tree is complete iff **every** directory is complete.
Clean and robust.

## 4. The reconciliation query (DB-only)

```sql
WITH scan AS (   -- expected: direct files per source directory, from the scan
    SELECT regexp_replace(sf.remote_path, '/[^/]*$', '') AS dir,
           COUNT(*)                       AS files,
           COALESCE(SUM(sf.file_size_bytes), 0) AS bytes
    FROM remote_sessions s
    JOIN remote_plans p            ON p.plan_id = s.plan_id
    JOIN remote_snapshot_files sf  ON sf.snapshot_id = p.snapshot_id
    WHERE s.session_id = %(sid)s
    GROUP BY 1
),
arch AS (        -- archived: direct files per directory, summed across bundles
    SELECT original_dir_path AS dir,
           SUM(direct_file_count)         AS files,
           SUM(direct_bytes)              AS bytes
    FROM directory_tree_index
    WHERE remote_session_id = %(sid)s
    GROUP BY 1
),
missing AS (     -- known omissions: source_missing files per directory
    SELECT regexp_replace(sf.remote_path, '/[^/]*$', '') AS dir,
           COUNT(*) AS files
    FROM remote_file_state st
    JOIN remote_plan_files pf     ON pf.plan_file_id = st.plan_file_id
    JOIN remote_snapshot_files sf ON sf.snapshot_file_id = pf.snapshot_file_id
    WHERE st.session_id = %(sid)s AND st.status = 'source_missing'
    GROUP BY 1
)
SELECT
    scan.dir,
    scan.files                         AS expected_files,
    scan.bytes                         AS expected_bytes,
    COALESCE(arch.files, 0)            AS archived_files,
    COALESCE(arch.bytes, 0)            AS archived_bytes,
    COALESCE(missing.files, 0)         AS source_missing_files,
    (COALESCE(arch.files,0) + COALESCE(missing.files,0) >= scan.files)
                                       AS is_complete
FROM scan
LEFT JOIN arch    ON arch.dir    = scan.dir
LEFT JOIN missing ON missing.dir = scan.dir;
```
Notes:
- Coordinate systems match: both `remote_snapshot_files.remote_path` and
  `directory_tree_index.original_dir_path` are **canonical source paths**. (Do
  **not** use `directory_archive_bundles.original_dir_path` — it stores the
  transient staging path, a known bug.)
- The result is `UPSERT`ed into `directory_completeness` (one row per dir).
- `>=` not `==`: dedup/re-archive can legitimately push archived slightly higher;
  a shortfall (`<`) is the real "incomplete" signal.

## 5. Components to implement

| Piece | Where | What |
|---|---|---|
| Migration `010_postgres_directory_completeness.sql` | `scripts/sql/` | the table + indexes; add to `_init_schema` migration tuple |
| `reconcile_directory_completeness(session_id)` | `src/pg_catalog.py` (or `pg_sessions.py`) | run the query above, UPSERT results, return `{dirs, incomplete}` |
| Session-complete hook | `src/remote_orchestrator.py` | after `update_remote_session(status='completed')`, call reconcile (best-effort; log on failure — never fail the archive on a reporting step) |
| `list_incomplete_directories(session_id=None)` | `src/pg_catalog.py` | report incomplete dirs + their expected/archived/missing gaps |
| CLI / inspector surface | `src/cli.py` / inspector | show a session's completeness summary + drill into incomplete dirs |
| Tests | `tests/` | (a) all archived -> complete; (b) archived + source_missing == expected -> complete; (c) genuine shortfall -> **not** complete; pure-SQL fixture or a stub-conn unit |

## 6. Edge cases / decisions

- **Local sessions:** the same idea applies but "expected" comes from the local
  scan, not `remote_snapshot_files`. Start with remote (the active use case);
  add local later with a `local_session_id` variant.
- **Multi-session directories:** a directory re-archived in a later session gets
  its own completeness row (keyed by session). Completeness is per-session-scan
  — "was *this run's* scan of the directory fully archived?"
- **Never block the archive:** reconciliation is a reporting step. Wrap it so a
  failure logs a warning and does not fail or roll back the archive.
- **Performance:** the scan side groups `remote_snapshot_files` (millions of
  rows) by computed parent dir — one-time per session, acceptable. Consider a
  functional index on `regexp_replace(remote_path,'/[^/]*$','')` only if it
  proves slow.
- **source_missing durability:** `remote_file_state` keeps `source_missing` rows
  after a chunk is `done` (only non-missing states are purged), so the missing
  count is available at session-complete time. Confirm before relying on it.

## 7. What this does and does not guarantee

- **Does:** for every directory, an automatic, durable flag that every scanned
  file was archived or is a recorded, accepted omission — entirely from the DB,
  no tape reads, no per-small-file catalog bloat.
- **Does not:** independently verify that each small file is physically readable
  on the tape (that would require reading the tape after writing — forbidden by
  the "No Independent Write Verification" rule). The trust chain for bytes on
  tape remains: scan -> fetch (per-file size verified) -> pack -> robocopy
  success -> bundle accounting. This feature closes the *scan-vs-archived
  accounting* gap, which is the real, DB-answerable question.
